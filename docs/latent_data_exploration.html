🔋 Receta 1: Batt_V → Corrección de Dead-Time del Inyector READY 🔴 ALTO IMPACTO

🧠 El problema

El voltaje de batería afecta el tiempo de apertura del inyector. Cuando baja el voltaje (ej: fan-on en el radiador), el solenoide del inyector tarda más en abrirse — el ECU cree que está inyectando X ms pero realmente está inyectando menos porque el inyector tardó más en responder.

Confirmado empíricamente: fan-on baja Batt_V −0.19V → shift de pw1 de +2.31%. Sessions VS interpreta esto como "sesión A es más rica que B" cuando es solo ruido de voltaje.

🧮 Fórmulas

pw_corrected = pw_raw + (dead_time(V_ref) − dead_time(Batt_V))
Donde la tabla dead_time(V) es:
V(V) | dead_time(ms) ─────|───────────── 14.0 | 0.80 13.5 | 0.85 13.0 | 0.92 ← V_ref 12.5 | 1.00 12.0 | 1.10 11.5 | 1.22 11.0 | 1.38 10.5 | 1.58
Fuente: Haltech/Link ECU injector dead-time data. Valores típicos para inyectores saturados pequeños (~20 lb/hr). Pendiente ~0.05-0.1 ms/V en rango 10-14V.
web/load_csv() en launch.py: agregar pw_corr = pw + dt_correction(Batt_V)

📂 Ruta de implementación

# 1. Crear tabla en ecu/injector_deadtime.py DEAD_TIME_TABLE = { 14.0: 0.80, 13.5: 0.85, 13.0: 0.92, 12.5: 1.00, 12.0: 1.10, 11.5: 1.22, 11.0: 1.38, 10.5: 1.58, } def dead_time_correction(batt_v: float) -> float: '''Return dead-time ms for given Batt_V (linear interpolation).''' voltages = sorted(DEAD_TIME_TABLE.keys()) if batt_v >= max(voltages): return DEAD_TIME_TABLE[max(voltages)] if batt_v <= min(voltages): return DEAD_TIME_TABLE[min(voltages)] for i in range(len(voltages)-1): v_lo, v_hi = voltages[i], voltages[i+1] if v_lo <= batt_v <= v_hi: t = (batt_v - v_lo) / (v_hi - v_lo) return DEAD_TIME_TABLE[v_lo] + t * (DEAD_TIME_TABLE[v_hi] - DEAD_TIME_TABLE[v_lo]) return DEAD_TIME_TABLE[13.0] # fallback # 2. Aplicar en load_csv() de launch.py (y f7.py) pw_raw = sf(r['pw1']) batt_v = sf(r.get('Batt_V', 13.0)) dt_corr = dead_time_correction(13.0) - dead_time_correction(batt_v) pw_corr = pw_raw + dt_corr # 3. dpw_eff ahora usa pw_corr en vez de pw_raw
Archivos a modificar: web/launch.py (load_csv), web/f7.py (_load_csv_rows), ecu/session.py

🌡️ Receta 2: humidity_pct + baro_hPa + MAT → Densidad del Aire Corregida READY 🟡 IMPACTO MEDIO

🧠 El problema

VDYNO usa `rho=1.10` fijo (Mexico City, ~940 hPa, 25°C). Pero la densidad del aire cambia con temperatura, presión, y humedad. El vapor de agua desplaza oxígeno molecular: a 30°C y 80% humedad, el aire tiene ~2% menos oxígeno que a 20%.

SAE J1349 no incluye humedad explícitamente. La fórmula extendida con humedad puede añadir ~0.5% de corrección en clima extremo — pequeña, pero elimina un confounder en comparaciones VDYNO entre días distintos.

🧮 Fórmulas (ISO 2533 + Magnus + SAE J1349 extendido)

# Paso 1: Presión de vapor de saturación (Magnus) p_sat = 0.61094 * exp(17.625 * T / (T + 243.04)) # kPa, T en °C # Paso 2: Presión de vapor real p_v = humedad_pct / 100 * p_sat # kPa # Paso 3: Densidad del aire húmedo rho = (baro * 100 / (287.058 * (T + 273.15))) * (1 - 0.378 * p_v / (baro / 10)) # Paso 4: SAE J1349 extendido con humedad P_d = baro / 10 - p_v # presión seca en kPa CF = 1.18 * (99 / P_d) * sqrt((T + 273) / 298) - 0.18 # Factor de humedad adicional: HF = 1 + 0.001 * (humedad_pct - 30) CF_total = CF * HF
T = MAT (manifold temp) o baro_temp_c si MAT no disponible.
baro = baro_hPa del sensor BMP280 (~1Hz en CSV).
humedad_pct = del sensor AHT20 (~8Hz en CSV).
rho actual: 1.10 fijo → con corrección: típicamente 1.05-1.18 según condiciones.

📂 Ruta de implementación

# 1. Crear web/air_density.py import math def saturation_vapor_pressure(t_c: float) -> float: '''Magnus formula, returns kPa. Valid for -40 to +50°C.''' return 0.61094 * math.exp(17.625 * t_c / (t_c + 243.04)) def air_density(baro_hpa: float, temp_c: float, humidity_pct: float) -> float: '''Humid air density in kg/m³.''' p_v = (humidity_pct / 100.0) * saturation_vapor_pressure(temp_c) p_dry = baro_hpa / 10.0 - p_v # convert hPa to kPa, subtract vapor return (p_dry * 100.0) / (287.058 * (temp_c + 273.15)) def sae_j1349_extended(baro_hpa: float, temp_c: float, humidity_pct: float) -> float: '''SAE J1349 correction factor with humidity extension.''' p_v = (humidity_pct / 100.0) * saturation_vapor_pressure(temp_c) p_dry = baro_hpa / 10.0 - p_v # kPa cf = 1.18 * (99.0 / p_dry) * math.sqrt((temp_c + 273.0) / 298.0) - 0.18 hf = 1.0 + 0.001 * (humidity_pct - 30.0) return cf * hf # 2. Modificar web/vdyno.py → _seg_physics() # Reemplazar: # rho = cfg['rho'] (1.10 fijo) # Con: # # rho from session env data (fallback to config default) # env = seg_env_stats(seg) # new fn to average baro/temp/humidity over segment # if env['baro'] and env['temp'] and env['humidity'] is not None: # rho = air_density(env['baro'], env['temp'], env['humidity']) # else: # rho = cfg['rho'] # fallback

🔄 Receta 3: gps_heading_rate → Análisis de Curvas (Cornering) READY 🟡 IMPACTO MEDIO

🧠 La oportunidad

Tenemos gps_heading_rate (°/s) y gps_turning en cada fila del CSV, pero nadie los usa. Con estos podemos calcular:
  • Radio de curva instantáneo: R = V / ω
  • Ángulo de inclinación estimado: θ = arctan(V² / (R × g))
  • Segmentación de curvas: entry → apex → exit
  • Comparación por sector: "este mapa acelera mejor en curvas?"
El DDFI2 es Alpha-N (TPS+RPM), y en curvas el TPS cambia constantemente → es donde más diferencias de mapa se notan. ¡Y justo es donde NO tenemos análisis!

🧮 Fórmulas

# Radio de curva instantáneo (metros) heading_rate_rads = gps_heading_rate * π / 180 R_curva = (gps_speed_kmh / 3.6) / heading_rate_rads # metros # Ángulo de inclinación (steady-state, sin IMU) lean_angle = atan((gps_speed_kmh / 3.6)² / (R_curva × 9.81)) # Clasificación de fase de curva: # Entry: heading_rate subiendo + speed bajando (frenado) # Apex: heading_rate máximo + speed mínimo # Exit: heading_rate bajando + speed subiendo (aceleración) # Segmentación de curvas consecutivas: # heading_rate cambia de signo = chicane # R_curva < 50m = curva cerrada # 50m < R_curva < 150m = curva media # R_curva > 150m = curvona
Limitación principal: GPS a 1-10Hz. Para análisis preciso de curvas se recomienda 10-25Hz. Nuestro GPS (u-blox NEO-6M) llega a 10Hz en modo binario. El ángulo de inclinación es estimado — no incluye contraviraje ni movimiento del piloto.

⚡ ⚠️ Receta 6 CORREGIDA: spark1 + spark2 → Spark Map Merge (NO hay sensor de knock) FACTIBLE 🟡 IMPACTO MEDIO

⚠️ CORRECCIÓN — NO hay sensor de knock en el DDFI2

Asumí incorrectamente que el DDFI2 tenía sensor de knock piezoeléctrico. Después de investigar el rtdata.xml completo y todo el protocolo:

❌ NO existe registro de knock sensor en el protocolo DDFI2
❌ NO existe knock retard register en RT_VARIABLES
❌ NO existe mención de knock en ningún flag, DTC, o campo

Este ECU no tiene capacidad de detección de detonación. spark1 y spark2 son simplemente el avance calculado por el ECU desde el mapa, sin ajuste por detonación.

🧠 Lo que SÍ podemos hacer con spark

Aunque no podemos inferir knock, spark1/spark2 siguen siendo datos útiles:
  • Comparar spark maps entre sesiones A vs B — merge REAL en vez de AVG
  • Detectar si la ECU limita avance en ciertas celdas (comparar spark_actual con el mapa base del eeprom)
  • Usar spark como proxy de carga — más avance = menos carga
  • Balance entre cilindros — spark1 vs spark2 revela diferencias front/rear
  • Detectar condiciones de seguridad — si la ECU retrasa spark globalmente (no por celda), puede ser safety override

Pero NO podemos detectar detonación. Las opciones reales para detectar knock sin hardware adicional son:
  • 🔴 Coil1_ADC / Coil2_ADC — feedback de bobina, pero no es ion sense
  • 🔴 Inj1_ADC / Inj2_ADC — feedback de inyector
  • 🟡 Chispa como indicador cualitativo (comparar avance entre sesiones)
  • 🟢 Plug chop (manual, no automatizable)
  • 🟢 EGT (requiere termopar en escape)

🧮 Fórmulas (lo que SÍ podemos hacer)

Spark adjustment from eeprom comparison: # Si spark_actual < spark_eeprom en > 3° en toda la tabla: # → La ECU está en modo seguridad (limp mode, sensor failure) # 3. Spark proxy for engine load: # A igual RPM×TPS, más avance = menos carga térmica # Si dpw_eff > 0 y spark1_B > spark1_A: cell B es más eficiente # 4. Cylinder balance: spark_balance = abs(spark1 − spark2) / mean(spark1, spark2) × 100 # > 10% de imbalance = problema en un cilindro (bobina, compresión, etc.)

🧭 Receta 10: IMU (MPU6050) — Acelerómetro + Giroscopio COMPLEJO 🟡 IMPACTO MEDIO (requiere HW + investigación)

🧠 La oportunidad — y la realidad

La idea original: Usar MPU6050 ($3) para detectar detonación por vibración (FFT) y medir ángulo de inclinación (lean angle). BL-DI-09 está planeado.

La realidad (investigación):
  • ❌ MPU6050 NO sirve para detectar knock. Frecuencia máxima 8kHz (muestreo ADC interno), la detonación del Thunderstorm (~88.9mm bore) está a ~6.4 kHz. Se necesita sampling a 20-40kHz (Nyquist) y un sensor piezoeléctrico dedicado.
  • ✅ Ángulo de inclinación SÍ es viable con MPU6050 usando filtro complementario (acelerómetro + giroscopio). Pero solo con GPS tenemos una aproximación.
  • ⚠️ El MPU6050 en un motor V-twin vibra mucho — necesita montaje aislado.
  • ⚠️ NO hay sensor de knock en el DDFI2 — ver Receta 6 corregida.

🧮 Fórmulas

Spark1/spark2 NO contienen información de detonación. # El MPU6050 tampoco sirve para detectar knock (frecuencia de sampleo insuficiente ~8kHz vs 40kHz+ necesario). # La única manera real de detectar detonación sin hardware dedicado: # 1. 🟢 Plug chop (manual, no automatizable) # 2. 🟢 Poscombustión audible (oído del piloto) # 3. 🔴 EGT (termopar en escape — requiere instalación) # 4. 🔴 Sensor piezoeléctrico dedicado con ADC 40kHz+ (investigación)

💡 HALLAZGO TRANSVERSAL

El DDFI2 NO tiene sensor de knock. Investigamos el protocolo completo (rtdata.xml de 328KB) y no hay ni un solo registro relacionado con detonación, knock, o retard de avance por knock. Las Recetas 6 y 10 han sido corregidas: lo que asumimos como "knock inference" no es posible. La detección de detonación en este sistema requiere hardware externo (EGT, sensor piezoeléctrico dedicado con ADC 40kHz+, o el oído del piloto).

En total: 8 recetas son SOLO SOFTWARE (10h total), 2 requieren hardware. La prioridad #1 sigue siendo Batt_V dead-time (+2.3% shift confirmado) y luego slope reference (GAP 4). El spark merge sigue siendo útil pero sin la dimensión de knock inference.
